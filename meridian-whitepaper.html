Meridian — White Paper

Machine-Native GIS Processing API v0.5.0 — March 2026

1. Executive Summary

Meridian is a spatial data processing API designed for AI agents and developers. It exposes 39 GIS endpoints — convert, reproject, validate, repair, clip, buffer, dissolve, tile, raster, PLSS lookup, and more — behind a frictionless micropayment rail. No accounts. No API keys. No subscriptions. An agent sends a request, receives a payment prompt, signs a small USDC transaction on Base, and gets its data back.

Meridian also exposes an Agent Skill on ClawHub, allowing OpenClaw agents to call it as a tool directly without managing payment manually.

2. The Problem

GIS data processing has a tooling gap at the agent layer.

Desktop tools (ArcGIS, QGIS) are designed for human interaction — GUIs, sessions, manual workflows. They do not compose cleanly into automated pipelines. Developer-facing libraries (GeoPandas, Shapely, GDAL) require Python expertise and environment setup. Cloud platforms (Esri ArcGIS Online, Google Maps Platform) are expensive, account-gated, and not designed for machine-to-machine use.

AI agents operating on geospatial data — converting formats, reprojecting layers, validating geometry, computing buffers — have no clean, lightweight, pay-as-you-go API to call.

Meridian is that API.

3. Design Principles

Machine-first. Every endpoint accepts multipart file uploads and returns processed files or JSON. No session state, no UI, no human steps. An agent can discover, call, pay, and receive results in a single automated loop.

Stateless payment. The payment model is designed for agents that don't have credit cards or user accounts. The 402/pay/retry loop means any agent with a funded Base wallet can use Meridian without pre-registration or human intervention.

No lock-in. Open standards throughout: GeoJSON, Shapefile, GeoPackage, KML. Output from any endpoint can flow into any GIS.

Composable. Endpoints chain naturally. Validate → repair → reproject → buffer → convert is a four-call pipeline. The batch endpoint runs up to 50 operations in a single request.

4. Payment Architecture

Meridian uses x402 with USDC on Base mainnet.

The flow:

  1. Agent calls any endpoint (no credentials)
  2. Server responds 402 Payment Required with x402 payment requirements (amount in USDC, Base network, recipient wallet address)
  3. Agent signs an EIP-3009 transferWithAuthorization payload with their Base wallet
  4. Agent retries with X-PAYMENT: <base64-encoded-payload> header
  5. Coinbase CDP facilitator verifies and settles transferWithAuthorization() on Base mainnet
  6. Request processes, result returns

Why Base / x402: Sub-second finality, negligible gas fees, EIP-3009 is a standard any EVM wallet supports. The Coinbase CDP facilitator handles verification and settlement — Meridian doesn't manage gas or RPC.

Pricing: $0.01 per MB, minimum $0.01, cap $2.00.

Anti-replay: Every x402 payment payload is verified and settled exactly once by the Coinbase CDP facilitator. Replay protection is handled at the facilitator level. Application-side idempotency records are maintained in PostgreSQL.

7. Agent Skill

Meridian publishes an Agent Skill on ClawHub. Any OpenClaw agent that installs the skill (clawhub install meridian) gains full knowledge of the API endpoint surface, payment flow, and tool descriptions — no manual setup required. The skill covers all 39 endpoints and handles the x402 payment negotiation automatically.

10. Why Meridian

AI agents that work with geospatial data have nowhere clean to send a file for processing. Meridian is the first GIS API designed specifically for that use case — stateless, machine-payable, composable, and accessible to any agent without human setup.

The Agent Skill layer takes it further: any OpenClaw agent can use Meridian as a native tool, making spatial data processing available in any agent workflow, without writing custom payment or API integration code.
